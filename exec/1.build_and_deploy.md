# 빌드 및 배포 문서

GitLab 소스 클론 이후 빌드 및 배포할 수 있도록 필요한 실행 환경, 빌드 환경 변수, 배포 특이사항, DB 접속 관련 설정을 정리한 문서입니다. 실제 비밀번호, 토큰, API Key 값은 문서에 기재하지 않고 운영 `.env` 또는 Jenkins credential에서 관리합니다.

## 1. 사용한 JVM, 웹서버, WAS, 제품 종류 및 버전

### Backend
* 언어: Java
* JVM: Java 21
* Framework: Spring Boot 4.0.5
* WAS: Spring Boot embedded Tomcat
* Build Tool: Maven

### Frontend
* 언어: TypeScript
* Framework: React 19
* Build Tool: Vite 8
* Package Manager: npm
* Web Server: Nginx

### AI
* 언어: Python 3.10 이상
* Framework: FastAPI
* ASGI Server: Uvicorn
* 주요 라이브러리: LangChain, boto3, pika, aio-pika

### CLI
* 언어: Python 3.10 이상
* Build Backend: hatchling
* CLI Framework: Typer
* 외부 스캔 도구: Trivy

### Engine
* 언어: Python 3.10 이상
* Framework: FastAPI
* ASGI Server: Uvicorn
* 용도: 업로드 파일 기반 스캔 처리 및 CLI rule/Trivy wrapping

### Infra / 배포
* 컨테이너: Docker, Docker Compose
* CI/CD: Jenkins Pipeline
* 이미지 저장소: Docker Hub
* 운영 도메인: `https://ssafer.co.kr`

### 개발 환경
* Backend: IntelliJ IDEA 또는 Java 21/Maven 지원 IDE
* Frontend: Visual Studio Code 또는 TypeScript/Vite 지원 IDE
* Python 모듈: Visual Studio Code 또는 Python 3.10 이상 지원 IDE

## 2. 빌드 시 사용되는 환경 변수

### Jenkins / Docker 이미지 빌드

Jenkins는 Docker 이미지를 빌드하고 Docker Hub에 push합니다.

* `DOCKERHUB_NAMESPACE`: Docker Hub namespace
* `DOCKERHUB_CREDENTIALS`: Docker Hub credential ID
* `IMAGE_TAG`: Docker image tag
* `SPRING_IMAGE`: Spring Backend image
* `FASTAPI_IMAGE`: AI FastAPI/Worker image
* `NGINX_IMAGE`: Frontend/Nginx image
* `ENGINE_IMAGE`: Scan Engine image
* `DEPLOY_ROOT`: Jenkins 서버 내 배포 파일 동기화 경로
* `EC2_2_SSH_CREDENTIALS`: EC2 #2 SSH credential ID
* `DEPLOY_EC2_2`: EC2 #2 배포 여부를 제어하는 Jenkins parameter

### Frontend 빌드 변수

Frontend/Nginx 이미지는 Vite build-time 환경 변수를 사용합니다.

* `VITE_API_BASE_URL`: Backend API base URL
* `VITE_GOOGLE_CLIENT_ID`: Google OAuth client ID
* `VITE_GITHUB_CLIENT_ID`: GitHub OAuth client ID
* `VITE_OAUTH_REDIRECT_BASE_URL`: OAuth redirect base URL

### Backend 빌드 및 실행 변수

Backend 빌드 자체는 Maven 설정을 사용하며, 운영 실행 시 아래 변수를 참조합니다.

* DB: `SPRING_DATASOURCE_URL`, `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD`
* Redis: `REDIS_HOST`, `REDIS_PORT`, `REDIS_PASSWORD`
* RabbitMQ: `RABBITMQ_HOST`, `RABBITMQ_PORT`, `RABBITMQ_USERNAME`, `RABBITMQ_PASSWORD`
* JWT: `JWT_SECRET`, `JWT_ISSUER`, `JWT_ACCESS_TOKEN_EXPIRES_SECONDS`, `JWT_REFRESH_TOKEN_EXPIRES_SECONDS`
* OAuth: `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, `GITHUB_OAUTH_CLIENT_ID`, `GITHUB_OAUTH_CLIENT_SECRET`
* S3: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `APP_SCAN_RAW_S3_BUCKET`, `APP_SCAN_RAW_S3_REGION`
* Email: `RESEND_API_KEY`, `EMAIL_FROM_ADDRESS`, `EMAIL_VERIFICATION_CODE_TTL_SECONDS`, `EMAIL_VERIFICATION_COOLDOWN_SECONDS`, `EMAIL_VERIFICATION_VERIFIED_TTL_SECONDS`
* Scan Engine: `APP_SCAN_ENGINE_ENABLED`, `APP_SCAN_ENGINE_URL`, `APP_SCAN_UPLOAD_SCAN_TIMEOUT_SECONDS`
* 내부 통신: `INTERNAL_TOKEN`

### AI / Worker 실행 변수

AI 서버와 worker는 EC2 #2 compose 환경에서 실행됩니다.

* LLM: `LLM_PROVIDER`, `LLM_EXPLAIN_MAX_TOKENS`, `LLM_FIX_MAX_TOKENS`, `LLM_VERIFY_MAX_TOKENS`
* GMS: `GMS_API_KEY`, `GMS_BASE_URL`, `GMS_MODEL`, `GMS_TEMPERATURE`, `GMS_TIMEOUT_SECONDS`, `GMS_FORCE_JSON_RESPONSE_FORMAT`
* Anthropic: `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`, `ANTHROPIC_TEMPERATURE`, `ANTHROPIC_TIMEOUT_SECONDS`
* Ollama: `OLLAMA_IMAGE`, `OLLAMA_CONTAINER_NAME`, `OLLAMA_VOLUME_NAME`, `OLLAMA_BASE_URL`, `OLLAMA_MODEL`
* S3: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, `APP_SCAN_RAW_S3_BUCKET`, `APP_ANALYSIS_RESULT_S3_BUCKET`
* RabbitMQ: `RABBITMQ_HOST`, `RABBITMQ_PORT`, `RABBITMQ_USERNAME`, `RABBITMQ_PASSWORD`, `RABBITMQ_VIRTUAL_HOST`, `AGENT_TASK_SCAN_REQUEST_QUEUE`
* Worker: `FASTAPI_BASE_URL`, `SPRING_BASE_URL`, `SPRING_API_SECRET`, `WORKER_HTTP_TIMEOUT_SECONDS`, `WORKER_MAX_CONCURRENCY`
* 선택 기능: `HASDATA_API_KEY`, `NVD_API_KEY`, `LANGSMITH_API_KEY`

### Engine 실행 변수

* `INTERNAL_TOKEN`: Backend와 Engine 사이의 내부 호출 인증 token
* `SCAN_MAX_FILE_COUNT`: 업로드 스캔 최대 파일 수
* `SCAN_MAX_UPLOAD_BYTES`: 업로드 스캔 최대 크기

## 3. 배포 시 특이사항

### 배포 서버 구성

EC2 #1은 서비스 서버로 사용합니다.

* Nginx
* Frontend 정적 파일
* Spring Backend
* PostgreSQL
* Redis
* RabbitMQ
* Scan Engine
* n8n

EC2 #2는 AI 분석 서버로 사용합니다.

* FastAPI
* AI Worker
* Ollama

### EC2 간 통신

* EC2 #1 Spring -> EC2 #2 FastAPI: `8000`
* EC2 #2 Worker -> EC2 #1 Spring callback: `8080`
* EC2 #2 Worker -> EC2 #1 RabbitMQ: `8989`
* 내부 서버 간 호출에 사용하는 token 값은 EC2 #1, EC2 #2, Engine/Worker 설정에서 일치해야 합니다.

### Jenkins 배포 흐름

1. GitLab 소스 checkout
2. Docker 이미지 빌드
3. Docker Hub push
4. EC2 #1 배포 파일 동기화
5. EC2 #1 docker compose 배포
6. AI 또는 EC2 #2 관련 변경이 있는 경우 EC2 #2 배포
7. Mattermost webhook으로 성공/실패 알림

### 운영 배포 주의사항

* 운영 `.env` 파일에는 secret, password, API key가 포함되므로 Git에 커밋하지 않습니다.
* Frontend OAuth client ID는 Vite build-time 값이므로 Nginx image 빌드 전에 준비되어야 합니다.
* Google/GitHub OAuth callback URL은 외부 서비스 콘솔에 로컬/운영 주소를 각각 등록해야 합니다.
* S3 presigned upload를 브라우저에서 직접 수행하므로 S3 bucket CORS 설정이 필요합니다.
* PostgreSQL, Redis, RabbitMQ는 compose healthcheck 통과 후 Spring이 시작되도록 구성되어 있습니다.
* Nginx는 Let's Encrypt 인증서 디렉터리를 읽기 전용으로 mount합니다.
* Docker Hub image tag가 Jenkins에서 생성한 tag와 compose 배포 변수의 tag가 일치해야 합니다.

## 4. DB 접속 정보 및 주요 계정/프로퍼티 파일 목록

### DBMS 정보

* DBMS: PostgreSQL
* 운영 컨테이너 이미지: `postgres:16-alpine`
* 운영 DB 컨테이너명: `ssafer-postgres`
* 운영 compose 기준 DB host: `postgres`
* 운영 compose 기준 DB port: `5432`
* DB명, 사용자, 비밀번호는 운영 `.env`의 `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`로 관리합니다.

### Backend DB 접속 설정

Spring Backend는 운영 compose에서 아래 변수를 통해 DB에 접속합니다.

* `SPRING_DATASOURCE_URL`
* `SPRING_DATASOURCE_USERNAME`
* `SPRING_DATASOURCE_PASSWORD`

PostgreSQL 컨테이너 초기화 변수:

* `POSTGRES_DB`
* `POSTGRES_USER`
* `POSTGRES_PASSWORD`

### 주요 계정 관리 방식

실제 계정명, 비밀번호, secret 값은 문서에 기재하지 않고 운영 서버의 `.env` 또는 Jenkins credential에서 관리합니다.

* DB 계정: `POSTGRES_USER`, `POSTGRES_PASSWORD`
* Spring DB 접속 계정: `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD`
* Jenkins credential:
  * Docker Hub credential
  * EC2 SSH credential
  * Mattermost webhook credential
  * EC2 #2 host credential
