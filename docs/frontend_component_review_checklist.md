# Frontend Component Review Checklist

## 목적

이 문서는 프론트엔드 PR 리뷰 시 컴포넌트 구조, 상태 설계, 재사용성, UX 품질을 빠르게 확인하기 위한 체크리스트입니다.

권장 사용 방식:

- PR 설명 작성 시 셀프 체크
- 리뷰어가 구조적 리스크 확인
- 리팩토링 필요 여부 판단

---

## PR 체크리스트

### 1. 구조와 책임

- [ ] 페이지 파일은 화면 조합 중심으로 작성되어 있다.
- [ ] API 호출과 복잡한 상태 흐름은 page가 아니라 hook 또는 feature 계층에 있다.
- [ ] 컴포넌트 이름이 역할을 명확히 설명한다.
- [ ] 하나의 컴포넌트가 너무 많은 책임을 동시에 가지지 않는다.
- [ ] 도메인 전용 컴포넌트는 `features/<domain>/components`에 배치되어 있다.

리뷰 포인트:

- page가 container 역할을 넘어서 business logic까지 떠안고 있지 않은가
- 파일 길이가 과도하게 길어졌다면 분리 기준이 있는가

### 2. 재사용성

- [ ] 반복되는 UI가 공통 컴포넌트 또는 도메인 컴포넌트로 추출되어 있다.
- [ ] 공통화가 과하지 않고, props가 지나치게 복잡하지 않다.
- [ ] 두 화면 이상에서 쓸 수 있는 패턴인지 고려했다.
- [ ] 하드코딩된 레이아웃/문구/variant가 불필요하게 반복되지 않는다.

리뷰 포인트:

- 재사용을 위해 억지로 추상화하지 않았는가
- 반대로 같은 코드가 여러 군데 복붙되지 않았는가

### 3. 상태 관리

- [ ] 원본 상태와 파생 상태가 구분되어 있다.
- [ ] 계산 가능한 값은 state가 아니라 계산으로 처리한다.
- [ ] `useEffect`는 외부 시스템 동기화에만 사용한다.
- [ ] 초기 로드, 새로고침, 재시도 로직이 불필요하게 중복되지 않는다.
- [ ] 버튼 중복 클릭이나 중복 요청 방지가 고려되어 있다.

리뷰 포인트:

- `useEffect -> setState`가 단순 계산용으로 쓰이지 않았는가
- loading / refreshing / submitting 상태가 분리되어야 할 곳에서 한 상태로 뭉개지지 않았는가

### 4. 커스텀 훅

- [ ] 커스텀 훅이 명확한 목적을 가진다.
- [ ] 반환값 이름이 읽기 쉽고 예측 가능하다.
- [ ] 데이터 조회, 액션, 파생값이 무질서하게 섞이지 않았다.
- [ ] 훅 내부의 에러/로딩 처리가 화면 요구사항과 맞는다.

리뷰 포인트:

- 훅이 단순 코드 이동만 된 것은 아닌가
- 재사용 가능한 단위로 잘 끊겼는가

### 5. API와 에러 처리

- [ ] API 호출은 도메인 API 파일에서 처리한다.
- [ ] 사용자 노출 메시지가 자연스럽고 일관된 한국어다.
- [ ] 공통 에러 래핑 규칙을 따르고 있다.
- [ ] 특수 에러 케이스만 분기 처리하고 있다.

리뷰 포인트:

- 컴포넌트에서 axios 에러 구조를 직접 해석하고 있지 않은가
- 동일한 실패 메시지가 파일마다 제각각이지 않은가

### 6. UI/UX

- [ ] 로딩 상태가 있다.
- [ ] 에러 상태가 있다.
- [ ] 빈 상태가 있다.
- [ ] 성공 후 피드백이 필요하면 토스트나 메시지가 있다.
- [ ] 키보드, 포커스, 모달 닫기 등 기본 상호작용이 고려되어 있다.

리뷰 포인트:

- 네트워크가 느릴 때도 사용자가 현재 상태를 알 수 있는가
- 작업 성공/실패 후 다음 행동이 명확한가

### 7. 접근성 및 문구

- [ ] 버튼과 입력 요소의 의미가 분명하다.
- [ ] placeholder만으로 의미를 전달하지 않는다.
- [ ] 상태 메시지가 지나치게 기술적이지 않다.
- [ ] 문구가 영문/한글 혼합으로 깨져 있지 않다.

리뷰 포인트:

- 시각적으로는 보이지만 의미상 부족한 요소가 없는가
- 사용자 문구가 개발자 시점 메시지로 노출되지 않는가

### 8. 코드 품질

- [ ] lint와 typecheck를 통과한다.
- [ ] 사용하지 않는 import, state, props가 없다.
- [ ] 주석은 필요한 곳에만 있고, 코드와 어긋나지 않는다.
- [ ] 파일/함수 이름이 현재 책임과 맞는다.

리뷰 포인트:

- “작동은 하지만 읽기 어려운 코드”가 남아 있지 않은가
- 향후 수정 시 다른 팀원이 쉽게 따라갈 수 있는가

## 빠른 리뷰 템플릿

아래 템플릿을 PR 코멘트나 리뷰 메모로 바로 사용할 수 있습니다.

```md
## Frontend Review

### Good
- page와 hook 책임 분리가 잘 되어 있습니다.
- 파생 상태를 별도 state로 저장하지 않고 계산으로 처리했습니다.
- 로딩/에러 상태가 사용자 관점에서 자연스럽게 보입니다.

### Needs Work
- page 파일 안에 아직 도메인 로직이 조금 남아 있습니다.
- 같은 API 흐름이 초기 로드와 새로고침에서 중복됩니다.
- 공통 컴포넌트로 추출 가능한 UI가 반복됩니다.

### Check Before Merge
- lint / typecheck 통과 여부
- 빈 상태와 실패 상태 문구 확인
- 버튼 중복 클릭 방지 여부 확인
```

## 셀프 리뷰용 질문

PR 올리기 전에 작성자가 스스로 아래 질문에 답해보면 좋습니다.

1. 이 컴포넌트는 정말 이 파일에 있어야 하나요?
2. 페이지가 너무 많은 로직을 직접 처리하고 있지 않나요?
3. 계산 가능한 값을 state로 들고 있지는 않나요?
4. 이 구조를 다른 팀원이 5분 안에 이해할 수 있나요?
5. 같은 패턴을 다음 화면에서도 재사용할 수 있나요?
6. 실패했을 때 사용자가 무엇을 해야 하는지 알 수 있나요?

## 현재 프로젝트 기준 중점 체크 항목

SSAfer 프론트에서는 아래 항목을 특히 중요하게 봅니다.

- `pages` 비대화 방지
- `features/*/hooks`로 도메인 로직 이동
- effect 안 즉시 `setState` 최소화
- 스캔/결과/프로젝트 화면의 중복 fetch 로직 방지
- 한글 문구 일관성 유지
- 토스트/오류 메시지 품질

---

## 권장 병합 기준

아래 조건을 만족하면 구조적으로는 병합 가능한 PR로 봅니다.

- lint 통과
- typecheck 통과
- page 책임 과다 없음
- effect 남용 없음
- 재사용 가능한 구조 유지
- 로딩/에러/빈 상태 구현
